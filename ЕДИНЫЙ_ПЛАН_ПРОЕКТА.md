## 1. Назначение проекта

CRM GoodPeople — платформа для кадрового агентства, автоматизирующая путь от первичной заявки клиента до подбора кандидатов, коммуникаций, календаря, документов и закрытия сделки.

Ключевая цель: убрать ручные разрывы между сайтом, менеджером, ботами, календарём, базой кандидатов, уведомлениями и договорами.

## 2. Цели системы

1. Автоматизировать сбор заявок с публичного сайта.
2. Запускать первичное вовлечение клиента через ботов и уточняющие сценарии.
3. Вести заявки менеджерами в CRM: статусы, канбан, карточки, заметки, история.
4. Согласовывать время собеседования через календарь CRM.
5. Создавать и отправлять ссылки на видеовстречи.
6. Подбирать кандидатов из базы по параметрам клиента.
7. Отправлять предложения соискателям и фиксировать ответы.
8. Генерировать договоры по шаблонам и отправлять клиенту.
9. Хранить персональные данные безопасно и готовить систему к требованиям 152-ФЗ.

## 3. Роли пользователей

| Роль | Задачи и права |
| --- | --- |
| Клиент | Оставляет заявку, общается с ботом, выбирает время собеседования, получает кандидатов и договор. |
| Менеджер | Ведёт заявки в CRM, управляет календарём, уточняет требования, запускает поиск кандидатов, отправляет договоры. |
| Соискатель | Хранится в базе, получает предложения через каналы связи, отвечает согласием или отказом. |
| Администратор | Управляет пользователями, скриптами бота, FAQ, шаблонами договоров, интеграциями и настройками. |

## 4. Основной пользовательский сценарий

1. Клиент заходит на сайт и заполняет форму: описание потребности и контакты Telegram, WhatsApp, MAX, Email.
2. Система сохраняет заявку в БД и уведомляет менеджера в CRM.
3. Бот пишет клиенту в доступные каналы, отправляет приветствие и уточняющие вопросы.
4. Бот предлагает свободные слоты из календаря CRM.
5. Клиент выбирает время, слот блокируется в календаре.
6. Менеджер видит встречу в CRM; система отправляет ссылку на видеосвязь.
7. Менеджер проводит интервью, уточняет требования и заполняет карточку заявки.
8. Менеджер запускает поиск соискателей.
9. Система показывает подходящих кандидатов; менеджер отправляет им предложения.
10. Соискатели отвечают, статусы обновляются в CRM.
11. Менеджер выбирает финальных кандидатов и отправляет клиенту.
12. Клиент подтверждает выбор.
13. Система генерирует договор по шаблону.
14. Клиент получает договор и подписывает его электронно или вручную.

## 5. Модули системы

### 5.1 Публичный сайт

- Лендинг с описанием услуг агентства.
- Форма заявки с описанием потребности и контактами.
- Экран подтверждения после отправки.
- Адаптивная вёрстка и SEO-база.
- Контентная часть может управляться через админ-раздел.

### 5.2 CRM-панель

- Канбан заявок.
- Карточка заявки с контактами, историей, заметками, статусами, календарём и документами.
- База кандидатов и импорт.
- Матчинг кандидатов.
- Уведомления менеджерам.
- Админ-раздел: пользователи, скрипты, FAQ, настройки.

### 5.3 Bot Engine

Каналы:
- Telegram через Telegram Bot API.
- WhatsApp через Green API.
- MAX через MAX Bot API.
- Email через SMTP или transactional email provider.

Логика:
- приветствие;
- уточняющие вопросы;
- отправка ссылок на внешние сервисы;
- выбор времени собеседования;
- подтверждения и напоминания;
- ответы на FAQ.

Администратор должен управлять сценариями бота без изменения кода.

### 5.4 Календарь и видеосервисы

- Менеджер задаёт свободные слоты.
- Клиент бронирует слот через бот или интерфейс.
- Система создаёт событие и блокирует время.
- Система создаёт или сохраняет ссылку на видеовстречу.
- Основные видеосервисы: Толк, Яндекс Телемост, Мост.
- Опционально: синхронизация с Google Calendar и Яндекс Календарём.

### 5.5 Кандидаты и матчинг

- Карточка кандидата: ФИО, возраст, специализация, опыт, зарплата, доступность, контакты.
- Импорт из текущей CRM через CSV/JSON.
- Поиск по параметрам заявки.
- Отправка предложений кандидатам через бот.
- Статусы предложений: отправлено, согласен, отказал.
- Отправка финальных кандидатов клиенту.

### 5.6 Договоры

- Шаблоны договоров с переменными.
- Автоподстановка данных из заявки и профиля кандидата.
- Генерация PDF из HTML-шаблона.
- Отправка договора клиенту через бот или email.
- Хранение версий договоров и истории изменений.
- Электронная подпись не входит в строгий MVP и планируется отдельным этапом.

### 5.7 Безопасность

- HTTPS/TLS для внешнего доступа.
- JWT-авторизация сотрудников.
- Роли admin и manager.
- Аудит действий менеджеров.
- Шифрование персональных данных на уровне БД как целевой пункт.
- Хранение данных с учётом требований 152-ФЗ.

## 6. Технологический стек

| Компонент | Технология | Назначение |
| --- | --- | --- |
| Frontend сайта | Next.js 14, React, TypeScript, Tailwind CSS | Публичный сайт и форма заявки. |
| Frontend CRM | Next.js 14, React, TypeScript, Tailwind CSS | Админ-панель, канбан, календарь, договоры. |
| Backend API | FastAPI, SQLAlchemy, Alembic | REST API, WebSocket, бизнес-логика. |
| Bot Engine | Python, aiogram/httpx | Telegram и мультиканальные сценарии. |
| Очереди | Celery, Redis | Фоновые задачи, уведомления, напоминания. |
| База данных | PostgreSQL 16 | Заявки, кандидаты, календарь, договоры. |
| Reverse proxy | Nginx | Прод-доступ, маршрутизация, SSL. |
| Инфраструктура | Docker Compose | Локальный и серверный запуск. |
| CI/CD | GitHub Actions | Проверки, миграции, build checks. |
| PDF | WeasyPrint или ReportLab | Генерация договоров. |

## 7. Ключевые API-направления

Заявки:
- `POST /api/applications`
- `GET /api/applications`
- `GET /api/applications/{id}`
- `PATCH /api/applications/{id}`
- `POST /api/applications/{id}/search`

Календарь:
- `GET /api/calendar/slots`
- `POST /api/calendar/slots`
- `POST /api/calendar/book`
- `POST /api/calendar/video-link`

Кандидаты и матчи:
- `GET /api/candidates`
- `POST /api/candidates`
- `POST /api/matches`
- `PATCH /api/matches/{id}`
- `POST /api/matches/{id}/send-to-client`

Договоры:
- `POST /api/contracts`
- `GET /api/contracts/{id}/pdf`
- `POST /api/contracts/{id}/send`

Настройки бота:
- `GET /api/bot/scripts`
- `PUT /api/bot/scripts`

Realtime:
- `GET /ws/applications/{id}`

## 9. Оставшийся backlog

### P1 — критично для продакшна

1. Выбрать или купить домен.
2. Настроить DNS: A-запись на сервер `62.181.53.36`.
3. Выпустить SSL-сертификат через Let's Encrypt.
4. Включить HTTPS-конфигурацию в `nginx`.
5. Проверить прод-доступ после HTTPS.
6. Формализовать регламент резервного копирования PostgreSQL.
7. Выполнить тест восстановления БД из backup.
8. Провести smoke-тест после включения HTTPS и каналов связи.

### P2 — каналы коммуникации

1. Подключить WhatsApp через Green API.
   - Переменные: `GREEN_API_INSTANCE`, `GREEN_API_TOKEN`.
   - Проверить отправку, получение и обработку ошибок.
2. Подключить MAX.
   - Переменная: `MAX_BOT_TOKEN`.
   - Проверить базовый сценарий сообщения и ответа.
3. Подключить Email/SMTP.
   - Переменные: `SMTP_HOST`, `SMTP_USER`, `SMTP_PASSWORD` и связанные настройки.
   - Проверить уведомления клиенту и менеджеру.

### P3 — следующий релиз

1. Электронная подпись договоров.
2. Дополнительные интеграции календарей.
3. Расширенные интеграции видеосервисов.
4. Усиленное шифрование персональных данных и аудит соответствия 152-ФЗ.
5. Платёжный модуль, если потребуется после MVP.

## 11. Smoke-тест после инфраструктурных изменений

Минимальный чеклист:
- сайт открывается по домену;
- HTTPS активен, браузер не показывает ошибок сертификата;
- `/api/health` отвечает успешно;
- форма заявки создаёт запись;
- менеджер видит заявку в CRM;
- бот получает задачу на коммуникацию;
- календарный слот можно выбрать и забронировать;
- уведомления доходят в подключённые каналы;
- договор можно создать и скачать;
- логи backend/frontend/bot не содержат критичных ошибок.

## 12. Backup и восстановление

Нужно зафиксировать:
- расписание backup PostgreSQL;
- место хранения backup;
- срок хранения;
- ответственного за проверку;
- команду восстановления;
- периодический тест восстановления на отдельной БД или staging-среде.

Критерий готовности: есть свежий backup и хотя бы один успешно выполненный тест восстановления.

## 13. Принятые технические решения

- WhatsApp: Green API.
- Хостинг: VPS + Docker Compose + Nginx.
- Авторизация CRM: login/password + JWT.
- Роли: admin и manager.
- Дизайн: корпоративный UI на Tailwind CSS.
- Скрипты бота: редактор в CRM.
- База соискателей: импорт из текущей CRM.
- Оплата: вне MVP.
- Видеосервисы: Толк, Яндекс Телемост, Мост.
- Договоры: PDF из HTML-шаблонов.
- Электронная подпись: отдельный этап после MVP.

## 14. Definition of Done для ближайшего релиза

Релиз можно считать готовым, если:
- `main` собран и задеплоен;
- домен и HTTPS работают;
- backend health endpoint доступен;
- заявки создаются с сайта;
- менеджер может обработать заявку в CRM;
- хотя бы Telegram-канал стабильно работает;
- дополнительные каналы либо подключены, либо явно выключены в конфигурации;
- backup PostgreSQL создан и восстановление проверено;
- smoke-тест пройден по чеклисту;
- секреты не хранятся в git;
- документация запуска и эксплуатации актуальна.
